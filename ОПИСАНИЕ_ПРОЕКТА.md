## 1. Технологический стек

| Слой | Технология | Зачем |
|---|---|---|
| Язык | TypeScript | Строгая типизация, меньше ошибок |
| Сервер | Node.js + Express | HTTP-сервер, REST API |
| База данных | PostgreSQL | Хранение всех данных |
| ORM | Drizzle ORM | Работа с БД через TypeScript-объекты (без сырого SQL) |
| Схема/валидация | Drizzle-Zod + Zod | Автоматическая генерация схем валидации из таблиц БД |
| Фронтенд | React 18 + Vite | Пользовательский интерфейс |
| Стили | Tailwind CSS | Утилитарные CSS-классы прямо в JSX |
| UI компоненты | Shadcn/UI | Готовые компоненты (кнопки, карточки, формы) |
| Маршрутизация (фронт) | wouter | Лёгкий роутер для React (аналог react-router) |
| Запросы к API | TanStack Query (React Query v5) | Кэширование, загрузка, инвалидация данных |
| Состояние корзины | Zustand | Глобальное хранилище корзины (без Redux) |
| Сессии | express-session + connect-pg-simple | Хранение сессий в PostgreSQL |
| Запуск (dev) | tsx (ts-node аналог) | Запуск TypeScript напрямую без компиляции |
| Сборка (prod) | Vite build | Минификация и сборка фронтенда |

## 3. База данных

Все таблицы описаны через Drizzle ORM в TypeScript — это называется **schema-first подход**: сначала описываешь таблицы кодом, затем командой `npm run db:push` Drizzle сам создаёт или обновляет таблицы в PostgreSQL.

### Таблица `sessions`
```
sessions
├── sid     (varchar, PRIMARY KEY) — уникальный ID сессии
├── sess    (jsonb)               — данные сессии в JSON (хранит userId)
└── expire  (timestamp)           — когда сессия истекает
```
**Назначение:** хранит активные сессии пользователей. Создаётся автоматически пакетом `connect-pg-simple`. Когда пользователь входит, здесь записывается его `userId`.

---

### Таблица `users`
```
users
├── id              (varchar, PRIMARY KEY) — случайный ID вида "abc123"
├── username        (varchar, UNIQUE)      — логин (уникальный)
├── password        (text)                 — пароль в открытом виде (*)
├── email           (varchar, UNIQUE)      — email
├── first_name      (varchar)              — имя
├── last_name       (varchar)              — фамилия
├── profile_image_url(varchar)             — ссылка на фото профиля
├── phone           (varchar)              — телефон
├── address         (varchar)             — адрес доставки
├── city            (varchar)             — город
├── birth_date      (varchar)             — дата рождения
├── created_at      (timestamp)            — дата регистрации
└── updated_at      (timestamp)            — дата последнего изменения
```
(*) В учебном проекте пароль хранится открыто. В реальном — нужно хэшировать через bcrypt.

---

### Таблица `products`
```
products
├── id           (serial, PRIMARY KEY) — автоинкрементный ID
├── name         (text)                — название товара
├── description  (text)                — описание
├── price        (integer)             — цена в КОПЕЙКАХ (например, 4990000 = 49 900 ₽)
├── image_url    (text)                — URL изображения (Unsplash)
├── category     (text)                — категория ("Электроника", "Аксессуары" и т.д.)
├── is_available (boolean, default=true) — есть в наличии или нет
└── created_at   (timestamp)           — дата добавления
```
**Важно:** цены хранятся в копейках, чтобы избежать проблем с дробными числами при расчётах. При отображении делится на 100 через функцию `formatPrice()`.

---

### Таблица `orders`
```
orders
├── id           (serial, PRIMARY KEY) — ID заказа
├── user_id      (text)                — ссылка на users.id (кто заказал)
├── total_amount (integer)             — итоговая сумма в копейках
├── status       (text)                — статус: 'pending', 'processing', 'completed', 'cancelled'
└── created_at   (timestamp)           — дата оформления
```

---

### Таблица `order_items`
```
order_items
├── id             (serial, PRIMARY KEY)
├── order_id       (integer)  — ссылка на orders.id
├── product_id     (integer)  — ссылка на products.id
├── quantity       (integer)  — количество штук
└── price_at_time  (integer)  — цена товара в момент покупки (в копейках)
```
**Зачем `price_at_time`?** Цены могут меняться, но история заказа должна показывать ту цену, по которой реально покупали. Поэтому цена фиксируется при создании заказа.

---

### Таблица `analytics`
```
analytics
├── id        (serial, PRIMARY KEY)
├── user_id   (varchar, nullable) — кто посетил (null если не авторизован)
├── page      (text)              — URL страницы, например "/catalog"
├── action    (text, nullable)    — тип события: 'view', 'click', 'purchase'
├── metadata  (text, nullable)    — дополнительные данные в JSON
└── timestamp (timestamp)         — когда произошло событие
```

---

### Связи между таблицами (Relations)
```
users ←── orders ←── order_items ──→ products
```
- Один пользователь может иметь много заказов
- Один заказ состоит из многих позиций (order_items)
- Каждая позиция ссылается на конкретный товар

Связи описаны через Drizzle `relations()` — это позволяет делать JOIN-запросы через ORM.

## 4. Бэкенд (сервер)

### `server/index.ts` — точка входа
Запускается командой `tsx server/index.ts`. Делает следующее:
1. Создаёт Express-приложение
2. Подключает парсинг JSON и URL-encoded данных
3. Добавляет middleware для логирования каждого запроса (метод, путь, время)
4. Вызывает `registerRoutes()` — регистрирует все API-маршруты
5. В режиме `dev` запускает Vite как middleware (фронтенд и бэкенд на одном порту)
6. В режиме `production` раздаёт собранный фронтенд как статику
7. Слушает порт 5000

---

### `server/db.ts` — подключение к базе данных
```typescript
const pool = new Pool({ connectionString: process.env.DATABASE_URL });
export const db = drizzle(pool, { schema });
```
Берёт строку подключения из переменной окружения `DATABASE_URL` и создаёт экземпляр Drizzle. Этот объект `db` используется везде для запросов.

---

### `shared/routes.ts` — контракт API
Здесь описаны **все маршруты** одновременно для сервера и клиента:

| Маршрут | Метод | Защита | Что делает |
|---|---|---|---|
| `/api/products` | GET | Нет | Список товаров (с фильтрами) |
| `/api/products/:id` | GET | Нет | Один товар по ID |
| `/api/orders` | GET | Да | Заказы текущего пользователя |
| `/api/orders/:id` | GET | Да | Один заказ по ID |
| `/api/orders` | POST | Да | Создать новый заказ |
| `/api/analytics` | POST | Нет | Записать событие аналитики |
| `/api/analytics/stats` | GET | Нет | Получить статистику продаж |
| `/api/login` | POST | Нет | Войти в аккаунт |
| `/api/register` | POST | Нет | Зарегистрироваться |
| `/api/logout` | POST | Нет | Выйти из аккаунта |
| `/api/auth/user` | GET | Нет | Данные текущего пользователя |
| `/api/user` | PATCH | Да | Обновить профиль |

Типы входных и выходных данных описаны через Zod-схемы — это даёт автодополнение в TypeScript и валидацию на сервере.

---

### `server/storage.ts` — слой работы с данными

Класс `DatabaseStorage` реализует интерфейс `IStorage`. Это **единственное место**, где пишется логика работы с базой данных. Маршруты не пишут SQL — они только вызывают методы storage.

**`getProducts(search?, category?, sortBy?)`**
- Строит запрос динамически: если передан поиск — добавляет `WHERE name ILIKE '%...%'`
- Если передана категория — добавляет `WHERE category = '...'`
- Сортировка: по дате (новые сначала), по цене (asc/desc)

**`createOrder(userId, orderData)`**
- Использует **транзакцию** (`db.transaction()`): сначала создаёт строку в `orders`, потом вставляет все строки в `order_items`. Если что-то упало — откатывает обе операции.
- Сразу возвращает заказ вместе с позициями (делает JOIN с products)

**`getSalesStats()`**
- Суммирует все заказы → `totalSales`, `orderCount`
- Считает суммы по категориям через JOIN с `order_items` и `products`
- Считает количество заказов по статусам

**`updateUser(id, userData)`**
- Обновляет поля профиля, автоматически проставляет `updatedAt = now()`

---

### `server/routes.ts` — регистрация маршрутов

Вызывается один раз при старте. Регистрирует маршруты на Express-приложении.

**Сидирование данных:** при первом запуске проверяет, есть ли товары в БД. Если нет — добавляет 6 тестовых товаров (Смартфон, Наушники, Кофемашина, Умные часы, Робот-пылесос, Рюкзак).

**Маршрут POST /api/orders (оформление заказа):**
1. Проверяет авторизацию через `isAuthenticated`
2. Берёт `userId` из сессии (`req.session.userId`)
3. Валидирует тело запроса через Zod-схему
4. Для каждого товара в заказе: загружает из БД, проверяет наличие, считает сумму
5. Создаёт заказ через `storage.createOrder()`
6. Записывает событие покупки в аналитику
7. Возвращает созданный заказ

## 5. Фронтенд (клиент)

### `client/src/App.tsx` — корень приложения

Оборачивает всё приложение в провайдеры:
- `QueryClientProvider` — даёт TanStack Query всем компонентам
- `TooltipProvider` — подсказки от Shadcn
- `Toaster` — всплывающие уведомления

Внутри — `Router` с `useAnalytics()` (логирует переходы) и набором маршрутов:
```tsx
<Route path="/" component={Home} />
<Route path="/catalog" component={Catalog} />
<Route path="/product/:id" component={ProductDetails} />
<Route path="/cart" component={Cart} />
<Route path="/checkout" component={Checkout} />
<Route path="/profile" component={Profile} />
<Route path="/auth" component={AuthPage} />
```

---

### `client/src/components/layout.tsx` — шапка и футер

Обёртка для всех страниц. Содержит:

**Шапка (header):**
- Логотип "МАГАЗИН" — ссылка на главную
- Навигация: Главная, Каталог (на десктопе)
- Мобильное меню (Sheet — боковое меню из Shadcn)
- Иконка пользователя → ссылка на профиль
- Иконка корзины с **бейджем** (количество товаров из Zustand)

Шапка "sticky" — прилипает к верху при прокрутке. Стиль `glass` — полупрозрачный фон с размытием (backdrop-filter).

**Футер:** колонки с навигацией, контактами, копирайтом.

---

### Страница: Главная (`home.tsx`)

**Hero-секция:**
- Большой заголовок с градиентным текстом
- Кнопка "Смотреть каталог" → `/catalog`
- Фото товара с декоративным blur-эффектом

**Популярные товары:**
- Загружает все товары через `useProducts()`
- Показывает первые 4 (`products?.slice(0, 4)`)
- Пока грузится — показывает скелетоны (анимированные заглушки)
- Карточки — компонент `<ProductCard />`

**Блок преимуществ:**
- 3 колонки: "Оригинальные бренды", "Быстрая доставка", "Поддержка 24/7"

---

### Страница: Каталог (`catalog.tsx`)

**Состояние:**
```typescript
const [search, setSearch] = useState("");
const [category, setCategory] = useState("");
const [sortBy, setSortBy] = useState("");
```
Все три состояния передаются в `useProducts({ search, category, sortBy })`. При каждом изменении TanStack Query автоматически делает новый запрос к серверу.

**Боковая панель фильтров** (FilterSidebar):
- Категории: Все, Электроника, Одежда, Дом, Аксессуары
- На десктопе — sticky sidebar слева
- На мобильном — скрыта, открывается кнопкой (Sheet из Shadcn)

**Поиск и сортировка:**
- Поиск — `Input` с иконкой лупы, фильтрует на лету
- Сортировка — `Select` выпадающий список

**Сетка товаров:**
- 1 колонка (мобайл) / 2 колонки (планшет) / 3 колонки (десктоп)
- Если ничего не найдено — пустое состояние с кнопкой "Сбросить фильтры"

---

### Страница: Товар (`product-details.tsx`)

1. Берёт ID из URL: `const { id } = useParams()`
2. Загружает товар: `useProduct(productId)` → `GET /api/products/:id`
3. Показывает: изображение, категория, название, цена, описание
4. Кнопка "В корзину" → вызывает `addItem(product)` из Zustand
5. Показывает тост-уведомление об успешном добавлении
6. Если `isAvailable = false` — кнопка задизейблена, текст "Нет в наличии"

---

### Страница: Корзина (`cart.tsx`)

Берёт данные из Zustand-хранилища (не с сервера!):
- `items` — массив `{ product, quantity }`
- `removeItem(id)` — удалить товар
- `updateQuantity(id, qty)` — изменить количество
- `getTotal()` — сумма всех товаров
- `clearCart()` — очистить всё

Если корзина пуста — показывает заглушку.

**Правый блок:** итого, кнопка "Перейти к оформлению" → `/checkout`

---

### Страница: Оформление заказа (`checkout.tsx`)

1. Проверяет авторизацию. Если не вошёл — предлагает войти.
2. Форма: ФИО, телефон, адрес доставки
3. Справа: минимёрный список товаров из корзины
4. При отправке формы: `createOrder.mutate({ items: [...] })`
   - На успех: очищает корзину, редирект на `/profile`
   - На ошибку 401: уведомление + редирект на `/auth`
5. После успешного создания — экран "Спасибо за заказ!" с кнопками

---

### Страница: Личный кабинет (`profile.tsx`)

**Шапка профиля (фон primary-цвет):**
- Аватар (первая буква имени или фото)
- Имя пользователя и логин (@username)
- Кнопка "Выйти"

**Левая колонка:**
- Форма редактирования профиля (имя, фамилия, телефон, город, адрес, дата рождения)
- Кнопка "Сохранить" → `PATCH /api/user`
- Виджет статистики продаж (из `GET /api/analytics/stats`)

**Правая колонка (2/3 ширины):**
- История заказов (`GET /api/orders`)
- Каждый заказ: номер, дата, сумма, статус (цветные бейджи), фото товаров

---

### Страница: Авторизация (`auth-page.tsx`)

Две вкладки (Tabs от Shadcn): "Вход" и "Регистрация"

**Вход:**
- Поля: логин, пароль
- `POST /api/login` → если успех, QueryClient обновляет кэш пользователя

**Регистрация:**
- Поля: имя, фамилия, логин, email, пароль
- `POST /api/register` → при успехе сразу входит

Если пользователь уже вошёл — автоматически редиректит на `/profile`.

## 6. Система аутентификации

Реализована **без внешних провайдеров** (убрана Replit OIDC). Всё хранится в собственной БД.

### Как работает сессия

```
Браузер                    Express-сервер              PostgreSQL
   |                            |                           |
   |--- POST /api/login ------->|                           |
   |   { username, password }   |                           |
   |                            |--- SELECT users --------->|
   |                            |<-- user record -----------|
   |                            |                           |
   |                            |  req.session.userId = id  |
   |                            |--- INSERT sessions ------->|
   |                            |                           |
   |<-- Set-Cookie: sid=xxx ----|                           |
   |                            |                           |
   |--- GET /api/auth/user ---->|                           |
   |   Cookie: sid=xxx          |--- SELECT sessions ------->|
   |                            |<-- { userId: "abc123" } --|
   |                            |--- SELECT users ---------->|
   |<-- { id, name, email } ----|                           |
```

### Middleware `isAuthenticated`
```typescript
(req, res, next) => {
  if (!req.session || !req.session.userId) {
    return res.status(401).json({ message: "Unauthorized" });
  }
  next(); // пропускает запрос дальше
}
```
Добавляется к защищённым маршрутам: `/api/orders`, `POST /api/orders`, `PATCH /api/user`.

### Cookie настройки
- `httpOnly: true` — JavaScript на клиенте не может прочитать cookie (защита от XSS)
- `secure: true` только в production — в dev работает по HTTP
- `sameSite: 'lax'` — защита от CSRF-атак
- `maxAge: 7 дней` — сессия живёт неделю

## 7. Корзина

Корзина реализована полностью на **клиенте** — никаких запросов к серверу.

### Хранилище Zustand (`hooks/use-cart.ts`)

```typescript
interface CartState {
  items: { product: Product, quantity: number }[]
  addItem(product)    // добавить или увеличить кол-во
  removeItem(id)      // удалить товар
  updateQuantity(id, qty)  // если qty=0 — удаляет
  clearCart()         // очистить всё
  getTotal()          // сумма в копейках
  getItemCount()      // общее количество штук
}
```

### Сохранение между сессиями
Zustand настроен с `persist` middleware + localStorage. Это значит, что если закрыть браузер и открыть снова — корзина сохранится.

### При оформлении заказа
1. Из корзины берётся `items[]`
2. Отправляется на сервер: `{ items: [{ productId, quantity }, ...] }`
3. Сервер сам проверяет цены из БД (нельзя подделать цену с клиента)
4. После успешного ответа — `clearCart()`

## 8. Аналитика

### Как работает логирование

В `App.tsx` вызывается хук `useAnalytics()`. Он следит за текущим URL через `useLocation()` и при каждом изменении отправляет запрос:

```
POST /api/analytics
{ page: "/catalog", action: "view" }
→ 204 No Content
```

Если пользователь авторизован — `userId` добавляется на сервере из сессии.

При оформлении заказа отдельно логируется:
```
{ page: "/checkout", action: "purchase", metadata: '{"orderId":5,"totalAmount":5580000}' }
```

### Статистика (`GET /api/analytics/stats`)

Вычисляется в `storage.getSalesStats()`:
- `totalSales` — сумма `total_amount` всех заказов
- `orderCount` — количество заказов
- `byCategory` — JOIN orders → order_items → products, группировка по `category`
- `byStatus` — количество заказов по каждому статусу

Показывается в виджете на странице профиля.

## 9. Полный путь пользователя по сайту

```
Открывает сайт → Главная (/)
    ↓ видит баннер + 4 популярных товара
    
Нажимает "Смотреть каталог" → Каталог (/catalog)
    ↓ ищет/фильтрует товары
    
Кликает на товар → Страница товара (/product/3)
    ↓ читает описание, нажимает "В корзину"
    ↓ товар добавляется в Zustand (localStorage)
    ↓ бейдж на иконке корзины увеличивается
    
Идёт в корзину (/cart)
    ↓ меняет количество, видит итого
    ↓ нажимает "Перейти к оформлению"
    
Страница оформления (/checkout)
    ↓ не вошёл? — предлагает войти
    ↓ вошёл? — заполняет форму, нажимает "Оплатить"
    ↓ POST /api/orders → транзакция в БД
    ↓ корзина очищается
    ↓ редирект на /profile
    
Личный кабинет (/profile)
    ↓ видит историю заказов с фото
    ↓ видит статистику продаж
    ↓ редактирует данные профиля → PATCH /api/user
```

```bash
npm run dev
```
